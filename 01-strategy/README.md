# 01 — Strategy

Get the problem honest before you commit to a solution. Frame the job, name the market, pick the position.

## What's in this phase

| File | Type | What it does |
|---|---|---|
| [framing-the-problem.md](framing-the-problem.md) | prompt | Forces a sharp problem statement before any solution talk |
| [jobs-to-be-done.md](jobs-to-be-done.md) | prompt | Reframes the product as the job the customer is hiring it for |
| [positioning-statement.md](positioning-statement.md) | prompt | Drafts the canonical "For [X], who [Y], our product is..." statement |
| [pestel-analysis.md](pestel-analysis.md) | prompt | Political / economic / social / technological / environmental / legal scan |
| [recommendation-canvas.md](recommendation-canvas.md) | prompt | Structures a recommendation with context, options, and decision criteria |

## Recommended order in this phase

1. **framing-the-problem.md** — get the problem statement honest first. Most strategy mistakes are downstream of a fuzzy problem.
2. **jobs-to-be-done.md** — translate the problem into the job. Forces customer language.
3. **positioning-statement.md** — write the position. If you can't, the problem or the job isn't sharp enough yet.
4. **pestel-analysis.md** — environmental scan. Use when the market is moving and you need to name what's changing.
5. **recommendation-canvas.md** — for documenting strategic recommendations to a team or board.

## When to skip

Strategy work has diminishing returns. If you already have a clear problem statement and you're confident about positioning, jump to `02-discovery/`. Don't write four strategy documents to delay talking to customers.
