# 04 — Launch

Put something testable in front of real traffic. Learn which message converts. Tell the story.

## What's in this phase

| File | Type | What it does |
|---|---|---|
| [visionary-press-release.md](visionary-press-release.md) | prompt | Amazon-style future-state press release for internal alignment |
| [storyboard-storytelling.md](storyboard-storytelling.md) | prompt | Narrative storyboard for the product story |
| [futuristic-product-faq.md](futuristic-product-faq.md) | prompt | Forward-dated FAQ pairing with the press release |
| [eol-product-message.md](eol-product-message.md) | prompt | Customer message for end-of-life / sunset of a product |
| [landing-page-experiment/](landing-page-experiment/) | **skill** | Generates three landing-page messaging variants + router + analytics dashboard |

## Recommended order in this phase

1. **visionary-press-release.md** — write the future-state press release before you launch. Use it as an internal alignment artifact.
2. **futuristic-product-faq.md** — pair the press release with an FAQ. Forces you to answer the hard questions a customer will ask.
3. **landing-page-experiment/** (skill) — generate the three messaging variants, set them live, and measure which value prop actually lands. At least 100 visitors per variant before drawing conclusions.
4. **storyboard-storytelling.md** — for the launch narrative on social, in a deck, or in a founder note.
5. **eol-product-message.md** — for when a product or feature is being retired.

## A note on sequencing

Most teams launch too late. The order above lets you launch the *messaging* before you launch the *product*: the landing-page experiment runs while the build is still happening, and the press release gets used internally to align the team, not externally on day one.

If you only do one thing in this phase, it should be the landing-page experiment. Real traffic on real variants tells you what to lead with on launch day.
